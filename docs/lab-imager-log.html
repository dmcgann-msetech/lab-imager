1. Project Vision

Initial Lab Imager concept and intended operational purpose.

Lab Imager is intended to be a full bench-lab microscope imaging workstation for soldering, electronics repair, PCB inspection, component diagnostics, and technical documentation.

The application will show the live HDMI microscope feed, allow screenshots, support annotations directly over the image view, provide zoom/view tools, and expose a right-side panel for notes, filenames, project data, and metadata.

Primary Use Case

WorkflowDescription
Electronics inspectionView solder joints, components, traces, bridges, cold joints, flux residue, damaged pads, and board defects.
Screenshot captureFreeze or capture microscope frames into timestamped image files.
AnnotationMark faults or observations using arrows, circles, rectangles, labels, and future measurement tools.
DocumentationSave images with notes and structured metadata for project history, class work, repairs, or engineering records.
Project identity: Lab Imager is not a generic camera app. It is bench instrumentation software for electronics diagnostics and lab documentation.

3. Technology Stack Decision

Final initial stack choice after reviewing C, C++, C#, and Python-style options.

Decision: Use C# / .NET with WPF as the first build target.

Reasoning

ChoiceAssessment
CToo low-level for the first pass. Useful for device drivers or specialized capture libraries, but inefficient for UI-heavy application development.
C++Powerful and appropriate for maximum low-level control, but unnecessary complexity for the first production path unless camera/capture constraints demand it.
C# / .NETBest balance for Windows-native UI, maintainability, installer support, settings storage, file handling, and future hardware integrations.

15. Initial Build Constraints and Safe Start Rules

Rules to keep the first implementation practical and stable.

Primary risk: overbuilding the application before proving the live capture pipeline.

Safe Start Rules

  1. Build the WPF shell first.
  2. Enumerate capture devices second.
  3. Render the live feed third.
  4. Capture PNG screenshots fourth.
  5. Add timestamped naming fifth.
  6. Only then add zoom, notes, annotations, and project records.

Do Not Start With

  • A complex database schema.
  • A full annotation engine.
  • A plugin system.
  • A cloud account model.
  • Measurement tools before calibration exists.
  • Video recording before still capture is stable.
Current next engineering move: create the repository, add the planned folder structure, commit this log, then build the first WPF application shell.

17. Engineering Viewport Reconstruction and UI Baseline Finalization

Full record of the Lab Imager viewport remediation cycle, final MainWindow.xaml rebuild, approved interface screenshot, and GitHub synchronization milestone.

Final status: The Lab Imager visual interface has been stabilized and approved as the working UI baseline for active application development. The final state was committed and pushed to origin/main under commit 3792a27 with message Finalize Lab Imager engineering viewport UI.

Purpose of This Log Entry

This entry records the complete work performed after the previous UI design approval section. The scope was not limited to cosmetic adjustment. The work became a full UI remediation and recovery cycle because the viewport grid, reticle layout, XAML mutation workflow, and file-editing method all required correction before the project could safely move into real application functionality.

The main objective was to convert the approved visual concept into a stable, buildable MainWindow.xaml implementation while preserving the existing WPF shell, custom titlebar, right-side Lab Session panel, and bottom status infrastructure.

Problems Identified During UI Review

ProblemObserved BehaviorImpact
Viewport grid scaleThe dotted grid appeared too large and heavy, reading visually as a pegboard pattern.Failed the approved engineering/microscope workstation aesthetic.
Reticle placementCorner markers appeared misplaced, too far apart, and not properly framing the center viewport focus area.Viewport did not read as an imaging or microscope targeting system.
Reticle directionSeveral bracket paths faced the wrong direction.Visual logic was incorrect because the reticles were not pointing inward toward the focal zone.
Patch workflowIncremental search/replace and regex edits repeatedly failed to reliably target the correct XAML regions.Created repeated rework cycles and inconsistent file state.
PowerShell here-string handlingLarge XAML here-strings did not close reliably during terminal paste operations.Produced partial writes and malformed XAML.
XAML validityMalformed content triggered WPF parser errors such as MC3000.Temporarily broke build integrity until the file was fully restored.

Recorded Build / Parser Failure Modes

Failure class: These were file-write and XML integrity failures, not .NET runtime logic failures. The application failed to build only when MainWindow.xaml was partially written or polluted with invalid wrapper text. 
MC3000: 'Data at the root level is invalid. Line 1, position 1.' XML is not valid.

MC3000: 'Unexpected end of file has occurred. The following elements are not closed.' XML is not valid.

Root Cause Assessment

Root CauseEngineering Finding
Oversized dot implementationThe dot-grid approach did not scale into the desired subtle viewport texture and remained visually too dominant.
Canvas-margin reticlesCanvas and margin-based positioning made the reticles fragile and difficult to align predictably.
Incremental patchingSmall patch commands were inappropriate for a layout-level XAML correction because the viewport was an interdependent visual structure.
Terminal-paste file replacementVery large here-string replacements were too error-prone for this file size and workflow.
Renderer wrapper contaminationMarkdown/code wrapper markers were mistakenly treated as part of file content during one attempted manual paste path.

Remediation Strategy Change

Standing rule established: Major Lab Imager UI/layout changes must use full-file replacement or direct editor overwrite workflows. Do not use manual find/replace hunts, regex surgery, or fragile incremental patching for substantial XAML layout work unless explicitly requested.

This rule was established because the UI work repeatedly proved that the stable engineering path is to replace the entire known-good file, validate with dotnet build, run the application, visually confirm, and then commit. This matches the preferred deterministic workflow and avoids repeated local corruption cycles.

18. Phase 2 UI Density, Camera Selector, Notes Toolbar, and Enumeration Foundation

Audit record for the Lab Imager Phase 2 interface refinement cycle, camera-selection workflow correction, DirectShow enumeration foundation, and pre-save-point stabilization.

Final status: The Lab Imager interface was refined from the oversized first-generation workstation shell into the approved compact widescreen engineering layout. The viewport remains dominant, the right-side Lab Session panel now includes camera-selection controls, the Notes area now has a two-row rich-text toolbar scaffold, and the top/bottom chrome has been reduced to a more professional software-instrument density.

Purpose of This Log Entry

This entry records the work completed after the approved engineering viewport baseline. The goal was to move Lab Imager from a static UI shell into a Phase 2-ready workstation interface while avoiding premature live-feed work. The user identified a critical design requirement before implementation went too far: the application must allow a camera to be selected and a preferred/default camera to be locked in. That requirement changed the Phase 2 design from simple camera detection to camera detection, camera selection, default-camera persistence, and graceful launch restoration.

The session also corrected the application scale. The original UI worked visually, but it read too large and kiosk-like. The revised design keeps the full widescreen window and large microscope viewport, but reduces title/status chrome, tightens right-panel controls, and gives the Notes workflow a proper toolbar foundation.

25. M2-B Overlay Engine Investigation — Native Preview Overlay Blocker

Engineering investigation record for the failed attempt to draw a true grid overlay above the active DirectShow preview surface.

Final status: True grid overlay over the current DirectShow IVideoWindow / WindowsFormsHost preview path is blocked by native rendering behavior. The current preview method remains valid for live video, but it is not the final architecture for overlay-heavy inspection workflows.

Root Cause

Root cause: The current M2-A preview path uses native DirectShow rendering through IVideoWindow hosted inside WindowsFormsHost. This gives a fast proof-of-camera path, but it creates airspace/rendering limitations that prevent reliable WPF or WinForms overlay drawing above the active video.

Engineering Decision

DecisionReason
Keep current DirectShow preview path temporarily.It successfully proves live camera rendering and remains useful as a working baseline.
Do not build final overlay tooling on top of IVideoWindow.The required grid, coordinate, measurement, and annotation layers need reliable overlay rendering.
Move overlay work to a new rendering architecture.A WPF-compatible frame pipeline or VMR9/windowless DirectShow path is required for true overlay support.
