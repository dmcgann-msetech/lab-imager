# Contributing to Lab Imager

## Overview

Lab Imager is a proprietary software project developed and maintained by MSE McGann Systems Engineering.

The project is publicly visible for development transparency, portfolio presentation, engineering documentation, testing, and demonstration purposes.

Lab Imager is not an open-source project.

---

## Contribution Policy

External contributions are not automatically accepted.

Pull requests, code submissions, feature proposals, architecture modifications, and documentation updates may be reviewed on a case-by-case basis.

Submission of a pull request does not imply acceptance.

All contribution decisions remain solely at the discretion of the project owner.

## Development Philosophy

Lab Imager is designed as a purpose-built engineering workstation focused on:

* Electronics inspection
* Microscope imaging
* Evidence capture
* Repair documentation
* Laboratory workflows
* Recording workflows
* Technical reporting

Changes should support those objectives.

Features that introduce unnecessary complexity, consumer-oriented functionality, or workflow dilution may be rejected.

---

## Engineering Standards

All accepted contributions should:

* Compile successfully
* Preserve existing functionality
* Avoid introducing warnings where practical
* Include validation steps
* Include documentation updates where applicable
* Follow existing project architecture patterns

Contributors should avoid:

* Broad search-and-replace operations
* Unvalidated architectural rewrites
* Large-scale UI redesigns without approval
* Direct modification of historical engineering records

---

## Documentation Requirements

When applicable, contributions should include updates to:

* README.md
* CHANGELOG.md
* RELEASE-NOTES.md
* roadmap.html
* lab-imager-log.html

Documentation should accurately reflect implemented behavior.

---

## Branching

Current development branch:

feature/wpf-frame-pipeline

Future branch strategy may evolve as the project matures.

Contributors should not assume direct commits to the primary development branch will be accepted.

---

## Validation Expectations

Before submission:

* Build succeeds
* Application launches
* Existing functionality remains operational
* No regressions are introduced
* Relevant workflows are manually tested

Where possible, provide:

* Validation notes
* Screenshots
* Reproduction steps
* Test results

---

## Bug Reports

Useful bug reports should include:

* Application version
* Windows version
* Camera device information
* Capture format information
* Steps to reproduce
* Expected behavior
* Actual behavior
* Screenshots where appropriate

---

## Feature Requests

Feature requests should describe:

* Problem being solved
* Proposed solution
* Workflow impact
* Expected user benefit

Preference is given to features that improve:

* Evidence workflows
* Imaging workflows
* Recording workflows
* Documentation workflows
* Engineering usability
